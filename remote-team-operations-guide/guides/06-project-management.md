# Project Management for Remote Teams

**Time to complete**: 2-3 weeks (setup + team adoption)
**Cost**: $7-15/user/month
**Prerequisites**: Communication platform, goals set
**Who owns this**: Operations lead or project manager

## Why Project Management Tools Matter for Remote Teams

**In-office project management**:
- Walk over to ask "What are you working on?"
- Whiteboard for planning
- See what's on people's desks
- Status updates in standups

**Remote requires**:
- ‚úÖ **Visible work** - Everyone sees what others are doing
- ‚úÖ **Async updates** - Status without meetings
- ‚úÖ **Centralized tasks** - Single source of truth
- ‚úÖ **Accountability** - Clear owners and deadlines

**The trap**: Using email, Slack, or spreadsheets for task management.
- ‚ùå Tasks get lost in threads
- ‚ùå No visibility into who's doing what
- ‚ùå Duplicated work
- ‚ùå Missed deadlines

**ROI**: Teams waste 5-10 hours/week searching for task information, clarifying priorities, and duplicating work.
- 10 people √ó 7 hours/week = 70 hours/week = **$182k/year wasted** (at $50/hour)
- PM tool cost: $10/user/month √ó 10 = $1,200/year
- **Net savings: $181k/year** (151x ROI)

## Phase 1: Setup & Configuration (Week 1)

### Day 1-2: Account Setup

**For Asana**:
1. Go to asana.com/create-account
2. Sign up with work email
3. Create workspace (company name)
4. Invite 2-3 pilot users (don't invite everyone yet)

**For other tools**: Similar signup process.

### Day 3-5: Structure Your Workspace

**The 80/20 project structure** (works for most 5-50 person companies):

**Tier 1: Teams** (by department or function)
- Engineering
- Sales
- Marketing
- Operations

**Tier 2: Projects** (within each team)
- Active projects
- Recurring work (ongoing operations)
- Backlog (future ideas)

**Example (Engineering Team)**:
- **Project**: Website Redesign
- **Project**: Mobile App MVP
- **Project**: Bug Fixes (recurring)
- **Backlog**: Future features

**Example (Sales Team)**:
- **Project**: Q1 Sales Pipeline
- **Project**: Lead Nurture Campaign
- **Recurring**: Weekly Sales Calls
- **Backlog**: New market expansion ideas

**Example (Marketing Team)**:
- **Project**: Q1 Content Calendar
- **Project**: Brand Refresh
- **Recurring**: Social Media Management
- **Backlog**: Future campaigns

### Day 6-7: Create Task Templates

**Task templates** ensure consistency and save time.

**Standard task template** (works for any task):
```
Task title: [Action verb] [What]
Example: "Design homepage mockup" not "Homepage"

Description:
- Objective: [Why are we doing this?]
- Context: [Background info or links]
- Acceptance criteria: [How do we know it's done?]

Assignee: [Person]
Due date: [Date]
Priority: High / Medium / Low
Tags: [Category, type]
Attachments: [Relevant files or links]
```

**Project template**:
```
Project name: [Descriptive name]

Sections:
1. Project Overview (description, goals, success metrics)
2. Planning (tasks for planning phase)
3. Execution (tasks for building/doing)
4. Review (tasks for QA, testing, review)
5. Launch (tasks for deployment/go-live)
6. Post-Launch (tasks for monitoring, iteration)
```

**Asana-specific**: Use Asana's built-in templates (File ‚Üí New ‚Üí From template).

**ClickUp/Notion**: Create template tasks/pages and duplicate for new projects.

## Phase 3: Best Practices & Workflows

### Task Management Best Practices

**1. Every task has an owner**
- ‚úÖ One person assigned (DRI - Directly Responsible Individual)
- ‚ùå Never assign to multiple people (diffuses responsibility)
- If multiple people work on it, one person is lead (owns completion)

**2. Every task has a due date**
- ‚úÖ Set realistic deadlines
- ‚ùå No due date = task drifts forever
- Use "today", "this week", "this month" if specific date isn't critical

**3. Task titles are action-oriented**
- ‚úÖ "Design homepage mockup" (clear action)
- ‚ùå "Homepage" (vague)
- Use verbs: Design, Build, Review, Launch, Research

**4. Break down large tasks**
- ‚úÖ Tasks should take 1-3 days max
- ‚ùå "Build mobile app" (too large)
- ‚úÖ "Design login screen", "Build authentication API", "Write tests for auth" (bite-sized)

**5. Use priorities**
- High: Urgent, blocks others
- Medium: Important, but not urgent
- Low: Nice to have, can wait

**6. Add context**
- Link to related documents (Notion, Google Docs)
- Add screenshots or mockups
- Explain "why" in description (not just "what")

### Project Management Best Practices

**1. Use sections/phases**
```
Project: Website Redesign

Sections:
- Planning (gather requirements, research)
- Design (mockups, feedback)
- Development (build pages)
- QA (testing, bug fixes)
- Launch (deploy, announce)
- Post-Launch (monitor, iterate)
```

**2. Set project milestones**
- Key checkpoints (e.g., "Design complete", "MVP shipped")
- Use Asana milestones or ClickUp milestones
- Celebrate when milestones are hit

**3. Use dependencies** (for complex projects)
- "Task B can't start until Task A is done"
- Asana, ClickUp, Monday support dependencies
- Prevents bottlenecks

**4. Recurring tasks** (for ongoing work)
- "Weekly sales report" (repeats every Monday)
- "Monthly invoicing" (repeats 1st of month)
- Automate instead of manually creating each time

**5. Assign time estimates** (optional)
- Helps with workload balancing
- "This task will take 4 hours"
- Some tools (ClickUp, Asana) support time tracking

### Sprint/Cycle Planning (For Agile Teams)

**If your team uses agile** (common for engineering teams):

**Workflow**:
1. **Backlog**: All future work (unplanned)
2. **Sprint Planning**: Pick tasks for next 1-2 weeks
3. **Current Sprint**: Work in progress
4. **Done**: Completed tasks

**Sprint cadence**: 1-2 weeks.

**Tools that support sprints**:
- Linear (built-in "Cycles")
- Asana (custom boards)
- Jira (native sprint planning)
- ClickUp (sprint view)

**Sprint planning meeting** (1 hour, every 1-2 weeks):
1. Review last sprint: What got done? What didn't?
2. Plan next sprint: Move tasks from backlog to sprint
3. Estimate capacity: How many tasks can team handle?
4. Assign tasks

**Daily standup** (async in tool):
- Each person updates task status
- Post in Slack or tool comments: "Yesterday: [tasks done] / Today: [tasks planned] / Blockers: [issues]"

## Common Workflows

### Workflow 1: Feature Development (Engineering)

**Steps**:
1. **Feature request** ‚Üí Created as task in backlog
2. **Planning** ‚Üí Refined, scoped, estimated
3. **Sprint planning** ‚Üí Moved to current sprint
4. **Development** ‚Üí Engineer builds, updates task status
5. **Code review** ‚Üí PR linked to task (Linear auto-links)
6. **QA** ‚Üí QA tests, comments on task
7. **Deploy** ‚Üí Feature shipped, task marked complete
8. **Post-launch** ‚Üí Monitor, iterate if needed

**Tool integration**:
- Linear ‚Üî GitHub (auto-link issues to PRs)
- Task status auto-updates when PR merges

### Workflow 2: Marketing Campaign

**Steps**:
1. **Campaign idea** ‚Üí Task created in backlog
2. **Planning** ‚Üí Define goals, audience, content plan
3. **Content creation** ‚Üí Tasks for each piece (blog, social, email)
   - Task: "Write blog post"
   - Task: "Design social graphics"
   - Task: "Schedule email in Mailchimp"
4. **Review** ‚Üí Manager reviews, provides feedback
5. **Publish** ‚Üí Content goes live
6. **Monitor** ‚Üí Track performance, iterate

**Tool**: Asana or Monday (visual boards work well for campaigns).

### Workflow 3: Sales Pipeline

**Stages**:
- Lead
- Qualified
- Demo Scheduled
- Proposal Sent
- Negotiation
- Closed Won / Closed Lost

**Tool**: Monday or Asana (board view for pipeline).

**Each deal** = Task with:
- Company name
- Contact person
- Deal size
- Expected close date
- Stage (drag to move between stages)

**Integration**: Connect to CRM (Salesforce, HubSpot) if you use one.

### Workflow 4: Bug Tracking

**Steps**:
1. **Bug reported** ‚Üí Task created in "Bug Backlog"
2. **Triage** ‚Üí Priority set (Critical / High / Medium / Low)
3. **Assigned** ‚Üí Engineer picks bug from backlog
4. **Fix** ‚Üí Engineer fixes, links PR
5. **QA** ‚Üí QA verifies fix
6. **Closed** ‚Üí Bug marked complete

**Tool**: Linear (built for bug tracking) or Asana.

## Common Pitfalls & Solutions

### Pitfall 1: "Tool is too complex, team doesn't use it"

**Symptom**: Team reverts to Slack/email, tool sits empty.

**Cause**: Overwhelming features, poor onboarding.

**Solution**:
1. **Start simple**: Use only basic features (tasks, due dates, assignees)
2. **Train slowly**: Introduce advanced features (automation, dependencies) after 1-2 months
3. **Make it required**: "All tasks in [Tool], not Slack"
4. **Provide templates**: Pre-configure projects so team doesn't start from blank slate

### Pitfall 2: "Tasks get created but never updated"

**Symptom**: Task status is always "In Progress", due dates ignored.

**Cause**: Team doesn't have habit of updating tasks.

**Solution**:
1. **Daily standup in tool**: Async updates in task comments
2. **Weekly review**: Manager checks task status in 1:1s
3. **Slack reminders**: Bot reminds people to update tasks
4. **Lead by example**: Managers update their own tasks religiously

### Pitfall 3: "Too many projects, can't find anything"

**Symptom**: Workspace is cluttered, search yields 100 results.

**Cause**: No structure, no archiving.

**Solution**:
1. **Use teams/folders**: Organize by department or theme
2. **Archive completed projects**: Move to "Archive" or "Completed" folder
3. **Naming convention**: "[Department] - [Project Name]" (e.g., "Eng - Mobile App MVP")
4. **Favorites/starred projects**: Pin important projects for quick access

### Pitfall 4: "Notifications are overwhelming"

**Symptom**: 50+ notifications/day, people mute tool.

**Cause**: Default notification settings are too noisy.

**Solution**:
1. **Adjust settings**: Notify only for @mentions and assignments (not all task updates)
2. **Use Slack for notifications**: Get notified in Slack instead of email
3. **Batch notifications**: Daily digest instead of real-time
4. **Mute noisy projects**: Unfollow projects you don't need to track closely
